# NEXUS BLOG — 技术补充指南

## 前言

这份文档是为不熟悉本项目中使用的具体技术的读者准备的。它不会深入代码细节，而是解释每个技术是做什么的、为什么选择它，以及它们如何协同工作。

---

## 一、项目是什么？

这是一个**个人技术博客**网站。你看到的网页和你与之交互的所有功能——浏览文章、注册登录、点赞收藏、评论排行——都是由下面介绍的这些技术组合实现的。

可以把它类比为一栋房子：Next.js 是房子的**框架结构**，React 是**房间里的家具**，SQLite 是**储物柜**，Tailwind CSS 是**装修风格**。

---

## 二、核心技术解释

### Next.js — 网站的"骨架"

**它是什么**: 一个基于 React 的 Web 开发框架。  
**通俗理解**: 如果说 React 是造房子的砖块，Next.js 就是设计图和施工队。它帮你决定哪些页面在服务器上提前生成（速度快），哪些在用户浏览器中动态渲染（内容实时）。  
**本项目中的应用**:
- 管理网站的所有页面路由（`/home`、`/posts`、`/settings` 等）
- 决定哪些页面提前渲染好（如文章列表），哪些实时生成（如管理后台）
- 提供 RSS Feed、站点地图等 SEO 基础设施

### React — 页面的"积木"

**它是什么**: Facebook 开发的 UI 库，用"组件"拼装网页。  
**通俗理解**: 像乐高积木。每个按钮、卡片、表单都是一个"积木块"（组件），可以反复拼接组合成页面。  
**本项目中的应用**: 所有的 UI 元素——文章卡片、评论框、导航栏、管理面板——都是 React 组件。

### TypeScript — 代码的"拼写检查"

**它是什么**: JavaScript 的增强版，在写代码时就检查类型错误。  
**通俗理解**: 像 Word 文档的拼写检查。你在写 `user.name` 的时候，它会提前告诉你"这个对象可能没有 name 属性"，避免上线后出错。

### SQLite — 数据的"小仓库"

**它是什么**: 一个轻量级数据库，数据存在单个文件里（`data/blog.db`）。  
**通俗理解**: 不需要安装额外的数据库软件，所有数据（用户、文章、评论、点赞）都在一个文件中。像一个便携的 Excel 表格，但功能强大得多。  
**为什么不用 MySQL/PostgreSQL**: 对于个人博客这种数据量不大、并发不高的场景，SQLite 足够且零配置。

### Drizzle ORM — 数据库的"翻译官"

**它是什么**: TypeScript 和 SQL 之间的桥梁。  
**通俗理解**: 你不用写 `SELECT * FROM users WHERE id = 1` 这种 SQL 语句，而是写 `db.select().from(users).where(...)` 这种 TypeScript 代码。Drizzle 把它翻译成 SQL，并且给你类型安全检查。

### Tailwind CSS — 页面的"化妆品"

**它是什么**: 一种直接在 HTML 中写样式的 CSS 框架。  
**通俗理解**: 传统方式是你写 CSS → 给元素加 class → 去 CSS 文件里找到这个 class 看它干了什么。Tailwind 反过来，class 名直接表达样式：`bg-red-500` = 红色背景，`text-lg` = 大号字体，`rounded-xl` = 圆角。  
**本项目自定义**: 在此基础上定义了赛博朋克风格的颜色变量（neon-cyan 青色、neon-purple 紫色等），以及扫描线、故障文字等特效。

### Framer Motion — 页面的"动画师"

**它是什么**: React 的动画库。  
**通俗理解**: 让网页元素有各种出场/退场/交互动画。本项目中的滚动入场效果、导航栏激活指示器、Toast 通知的滑入滑出都是它做的。

### JWT (JSON Web Token) — 用户登录的"通行证"

**它是什么**: 用户登录后服务端签发的一个加密"令牌"。  
**通俗理解**: 就像酒店房卡。你登录成功后，浏览器拿到一张"房卡"（存在 Cookie 里），之后每次请求都自动带上这张卡，服务端一看就认出你是谁了。卡的有效期是 7 天。  
**安全加固**: Cookie 设置了 `httpOnly`（JavaScript 无法读取）和 `SameSite`（跨站请求不发送），降低被窃取的风险。

### Nodemailer — 邮件发送的"邮递员"

**它是什么**: Node.js 的邮件发送库。  
**通俗理解**: 注册时"发送验证码"这一步，就是它通过 QQ 邮箱 SMTP 服务，把 6 位验证码邮件投递到你的邮箱。

### ECharts — 管理后台的"图表引擎"

**它是什么**: Apache 开源的图表库。  
**通俗理解**: 管理后台仪表盘里的流量曲线图、互动柱状图都是用 ECharts 画的。  
**优化处理**: ECharts 体积较大（~1MB），所以只在你打开管理后台时才加载，不影响普通读者浏览博客。

### Fuse.js — 搜索的"模糊匹配器"

**它是什么**: 轻量级模糊搜索库。  
**通俗理解**: 你在搜索框输入关键词，它不要求完全匹配，而是算法计算出哪些文章最相关。比如搜"docker部署"能找到标题是"Docker 容器部署指南"的文章。  
**优化处理**: 输入停止 300 毫秒后才执行搜索，避免每次按键都触发搜索造成卡顿。

## 三、系统如何协同工作

从用户视角出发，几个典型流程：

### 浏览文章
1. 你访问 `/home` → Next.js 从 SQLite 和 MDX 文件读取文章列表 → 提前渲染成 HTML → 浏览器直接显示（ISR 缓存策略，1 分钟后刷新）
2. 点击一篇文章 → Next.js 读取文章内容 → 渲染页面（包含 LikeButton、BookmarkButton、CommentSection 等 React 组件）
3. 你点赞 → 浏览器发送请求到 `/api/likes` → SQLite 更新数据 → 页面显示新的点赞数（乐观更新，先改 UI 再确认）

### 注册登录
1. 输入邮箱 → 点击"发送验证码" → Nodemailer 发送邮件 → SQLite 记录验证码（5 分钟有效，60 秒内不能重复发送）
2. 输入用户名、密码、验证码 → 提交到 `/api/auth/register` → 密码由 bcrypt 加密存储 → 签发 JWT → 存入 Cookie
3. 登录后 → Header 组件显示你的头像/用户名 → 每次 API 请求自动携带 JWT

### 发布文章
1. 点击"发布文章" → MarkdownEditor 加载 → 如果之前有草稿，从 localStorage 恢复
2. 用工具栏插入 Markdown 格式 → 切换预览模式查看渲染效果 → 自动保存草稿
3. 发布 → 保存到 SQLite → 清除缓存 → 跳转到"我的文章"

### 管理后台
1. 只有 admin/super_admin 角色的用户能看到"管理后台"入口
2. 仪表盘 → 动态加载 ECharts → 从 SQLite 聚合查询统计数据 → 渲染图表
3. 用户管理、评论管理、审核改名等操作 → PATCH/DELETE API → 实时生效

## 四、关键概念速查

| 术语 | 一句话解释 |
|------|-----------|
| ISR (增量静态再生成) | 页面提前生成，但在一定时间后自动更新 |
| SSR (服务端渲染) | 每次请求都在服务器上生成页面 |
| SSG (静态生成) | 构建时就生成好，永远不变 |
| CSR (客户端渲染) | 页面先加载一个空壳，然后在浏览器中填充数据 |
| JWT | 加密的用户身份令牌，存在 Cookie 中 |
| ORM | 用代码操作数据库，而不是写 SQL |
| WAL (Write-Ahead Logging) | SQLite 的性能优化模式，读写并发不阻塞 |
| OG (Open Graph) | 网页分享到社交媒体时的标题、描述、图片 |
| RSS | 博客的订阅源，读者可以用 RSS 阅读器订阅 |
| httpOnly Cookie | 浏览器无法用 JS 读取的 Cookie，更安全 |
| Route Group | Next.js 中不影响 URL 的目录分组方式 |
| Dynamic Import | 按需加载代码，而不是一次性加载所有代码 |
| Debounce (防抖) | 等用户停止操作一段时间后再执行，避免频繁触发 |
| Optimistic Update | UI 先显示操作成功的状态，后台再确认 |

## 五、部署与运维

### 服务器信息
- **IP**: 43.138.158.121
- **进程管理**: PM2 (Node.js 进程守护)
- **环境变量**: `.env.local` 文件 (JWT_SECRET, SMTP 配置)

### 日常维护
- 新增文章: 直接在 `content/posts/` 目录下添加 `.md` 文件即可
- 系统重启: `pm2 restart my-blog`
- 查看日志: `pm2 logs my-blog`
- 数据库备份: 复制 `data/blog.db` 文件
